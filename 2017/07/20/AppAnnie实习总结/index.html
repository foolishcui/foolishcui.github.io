App Annie 实习总结

Jul 20, 2017

2017上半年一直在北京 AppAnnie 公司做交互设计的实习生，受益匪浅：

1）这段实习经历首先让我对公司里专业的交互设计流程有了一个全面的了解；
2）由于平日的主要工作内容是为公司复杂的产品线协助建立组件库，这在公司内部被称为 Component Libary, 我们在建立组件库的过程中也是困难重重；
3）积累了为业务逻辑复杂的网站开发交互组件的知识；
4）积累了许多日常工作的经验教训。

主要工作内容

1）协助团队创建一套组件库文档，并以这套文档作为基础为不同组件制定设计规范。在这个过程中需要确定哪些组件应该被放进组件库中，并把这些组件进行分类整理；同时确定哪些是旧有的组件，而这些旧有的不符合规范的组件则会被新组件逐渐替代。
2）由于公司的主要业务是向用户展示数据，Table很自然的成为了产品中使用最普遍的组件，也承载了复杂的业务逻辑，为保证Table组件在产品中行为与样式的一致性，我们决定首先为Table制定一套组件使用规范。
3）我们找遍了页面中所有出现的Table组件，为Table制定了一套设计设计规范，包含元素、样式、行为、数据展示方法等，这套规范将被用在未来开发新产品时使用，同时也用于修改旧有的不符合规范的Table组件。
4）之后为保证公司内所有设计师都能够正确的使用Table组件，我们为Table建立了Sketch文件方便设计师开发新产品时使用，同时制作了网页原型交由工程师开发使用。
5）期间Airbnb推出了一套叫做Airbnb/React-Sketchapp的插件，可以帮助把React代码渲染到Sketch文件中查看，我们尝试使用这个工具开发组件库，以缩小设计与开发之间的沟通成本。但由于该插件所支持的API数量有限，不足以支持现有网站中复杂的组件开发，我们最终放弃了这个方案。
6）后来团队开始了Filters组件与Date Rang Picker两个组件的重设计与设计规范的制定。这也是团队在建立组件库最大的难点所在，我们需要先对不合理的交互组件进行重设计从而制定全新的设计规范。

以上总结了我在 AppAnnie 公司所参与的工作，这个过程中让我认识到了为复杂的产品建立一套组件库的复杂性，设计工作中沟通的重要性以及一套科学合理的设计方法的重要性。我想每一个公司在高速成长的过程中都要经历为产品建立一套组件库的过程，这个过程可能非常困难，需要权衡各方面的利益，但完成之后将会有极大帮助。

为什么需要组件库

公司在快速扩张时，会不断面临新的产品需求，就会不断为产品中添加新的交互组件来满足需求，此时一套详细的组件库与组件规范能够：

  • 合理的复用现有组件，加快开发周期，创造协调一致的用户体验。
  • 迭代旧有组件，与现有组件保持风格一致
  • 不同设计师的设计方案能够保持类似的设计风格
  • 维护多平台体验的一致性

因此组件库是把现有的产品组件的样式、行为、数据模式收集起来，为常见的设计问题提供可循环复用的解决方案。当有了这样一套组件库后，产品设计与开发工作都中要遵循组件库中的通用规则，从而保证了用户体验的一致性、代码的可复用性以及设计团队日常工作的可参照性。

如何为业务逻辑复杂的产品建立组件库

我们也在不断探索科学合理的的方式去为 AppAnnie 这样具有复杂业务逻辑的产品建立组件库：

1）我们需要不断考虑哪些组件应该被放进组件库中去重复使用，而哪些组件需要被淘汰，从而制定出一个组件库的索引文档；
2）我们需要定义每个组件的功能、行为与使用场景；
3）我们需要制定一套统一的产品设计风格，包含字体排版，颜色，图标，间距，信息架构。把设计风格统一下来能够帮助设计师在独立探索设计方案的时候有一个统一的方向。
4）我们需要为组件库中的每个组件制定设计规范，对于在产品中频繁使用的组件，如Table, Filters等，需要设计团队投入更多的精力去思考调研，得到更为强大有效的解决方案。这个过程中同时需要找出组件在产品中所有的使用场景，确保新制定的规范能够满足所有业务逻辑。
5）我们需要确定不同组件设计规范制定的先后顺序，对于频繁使用或开发新功能所涉及到的组件需要被提前制定。
6）我们需要与开发团队充分沟通确保最新制定的组件设计规范能被顺利实现。
7）在确定了正确合理的设计规范后我们会制作Sketch原型供设计师使用，同时也会制作Axure原型或HTML高保真原型用作用户测试。

组件库索引文档的制定

组件库的索引文档包含了产品中可复用的所有组件，为日后具体组件设计规范的制定奠定基础。

我们采用了如下的分类方式对组件库索引文档进行整理，合理的分类方式能够帮助产品、设计与开发团队更有效的理解组件库：

  • Style: 定义了产品中最基本的视觉样式，包含字体排版、颜色、图标等；
  • Pattern
    • Data Pattern: 定义了产品中数据的展示形式
    • Layout Pattern: 定义了组件的布局形式
  • Behaviour
    • UI Controls: 定义所有的通用组件
    • WidgetsL: 定义与产品本身业务逻辑结合密切的组件
    • System Behaviour: 定义产品中应该出现的标准的交互行为，往往伴随一个固定的业务逻辑

有了这样一套组件库索引文档作为基础，我们开始了具体组件设计规范的制定。

具体组件设计规范的制定

当有了一套组件库索引文档之后，我们需要站在组件复用的角度为不同的组件制定详细的设计规范。

在这个过程中我们首先把一个组件在产品中所有场景下的使用截图进行整理，找出所有导致用户体验不一致的细节并总结到一个文档中，并一一改进找到合理的解决方案，然后在这基础上建立一套设计规范。之前的文档也会保留用于日后旧组件的迭代需要。

我们对组件设计规范中的内容采用了如下的分类方法：

  • Component Usage: 定义组件的应用场景
  • Elements: 定义组件中所出现的元素以及元素的含义
  • Style: 定义组件的视觉样式
  • Component Behaviour: 定义组件的行为
  • Related Component Behaviour: 定义其他相关组件的交互行为
  • Data Pattern: 定义组件中数据的展示形式

这样的分类方法贯穿在了所有组件规范的制定的思考过程中。

合适的设计流程

一个合理的交互设计流程奠定了开发一个新产品或新功能的基础，保证了产品设计的质量。我想任何一个团队都需要根据自己产品的设计特点与团队人员配置，确定合适的设计流程。

在产品的初期设计过程中，非常重要的一点是产品经理、设计师与开发人员之间的沟通。一是为了确保设计师能够正确理解产品需求，二是为了确保设计能够被开发人员成功实现。有效的沟通应该贯穿在整个产品设计过程中，这样确保了产品设计以最大的效率向正确的方向发展。

在交互设计的交付物中，一份清晰整洁的交互文档是必要的，可以帮助团队人员清晰的理解设计目标、组件的逻辑、不同状态的反馈以及尺寸规格。除此之外，一份原型可以更好的帮助团队人员理清逻辑，这份原型可以使用任何工具，Axure、inVision、Marvel、Flinto、Framer与墨刀都是非常不错的工具。

设计师之间的沟通也是非常必要的，这样可以方便设计师了解彼此正在做的项目，对于设计难点可以拿出来一起讨论，同时也可以确保产品保持统一的设计风格与类似的交互逻辑。这同时也是组件库的意义所在。

为业务复杂的网站开发交互组件

如公司另外一位设计师所言：通用组件的设计往往是最难的，因为在设计过程中容易被固有UX Pattern限制住，导致不能解决产品的特有需求。

在App Annie实习的最后一段时间，做了通用组件Date Range Picker的重设计，它可以帮助选择图表所要展示的日期区间，以及选择数据粒度(Day,Week,Month)。重设计工作中，我做了充分的UX Research去查看竞品网站如何设计，并结合业务需求仔细分析了现有组件的问题，最后做出proposal并在Axure中制作了高保真原型。

工作经验教训

除了知识积累，这次实习经验给我留下最宝贵的是工作中的经验教训：

  • 设计工作要重视数据的重要性，用数据来论证设计。
  • 踏踏实实、有耐心的做事情，设计工作并非永远充满趣味，总会有枯燥无聊的时候，但这也是工作内容的重要部分。
  • 每次会议之前想好要分享的内容，保证自己为会议贡献出自己有价值的想法。
  • 合理的安排好工作时间，什么时间做出什么成果要有一个大体的预期。
  • 沟通在设计工作中是非常重要的，同时也要注重沟通的高效率。
  • 一个非常大的项目往往可以分割成许多细小的工作，这样容易下手。
  • 在工作之余要保持学习，把新鲜的知识、工具引入到工作中。
